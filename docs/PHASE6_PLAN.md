## 3. Binding Decisions

These pin the spec so Step A has no ambiguity. Each deviation from `docs/PLAN.md` §10 is called out. **If you disagree with any of these, say so before Step A starts.**

### 3.1 Layout: 3-pane desktop, single-column on mobile

- **Desktop (≥ 1024 px, the `lg:` Tailwind breakpoint)**: a 3-pane grid. Left rail **256 px** wide (`<FolderSidebar>`), middle **320 px** wide (`<NoteList>`), right **flex-1** (`<NoteDetail>` or `<NoteEditor>`). Header stays at the top (theme toggle + user menu), outside the grid.
- **Tablet (768–1023 px)**: 2-pane — left rail collapses to a `<Sheet>` (shadcn) triggered by a hamburger in the header. Middle + right stay.
- **Mobile (< 768 px)**: single column with route-driven views. The folder sidebar becomes a top-of-page "folder pill" that opens a `<Sheet>`. The note list and the detail are separate routes (`/`, `/notes/:id`).
- Rationale: matches Notion / Apple Notes mental model; 3-pane is the conventional shape for a notes app. Mobile-first apps with the same complexity use route-driven views because a 256-px sidebar + 320-px list is too cramped below 768 px.

### 3.2 Folder/tag sidebar: flat list, not a tree

- The sidebar shows, in order:
  1. **"All notes"** — pseudo-folder, count = total notes (computed in the UI from `notesApi.list`); active when no folder or tag is selected.
  2. **"Unfiled"** — pseudo-folder, count = notes with `folderId === null`; active when selected.
  3. **Folder list** — one entry per `foldersApi.list` row, sorted by name, count = `noteCount` (server-computed per `FolderService.cs:29`); active when its `id` is in `?folderId=…`.
  4. **Tag list** — one entry per `tagsApi.list` row, sorted by name; active when its `id` is in `?tagId=…`. (Tag count is **not** in `TagResponse`; the sidebar shows the tag name without a count to avoid a per-tag round-trip; this is consistent with `docs/PLAN.md` §10 which doesn't promise tag counts.)
  5. A footer button **"New folder"** that opens an inline `<Dialog>` (name + "Create").
- Each folder row has a hover-revealed `trash` icon → opens a `<AlertDialog>` ("Delete folder? Notes will be unfiled, not deleted." → confirm).
- No drag-to-reorder; no nesting; no per-folder "edit name" UI (the spec says CRUD; rename is a future enhancement).
- Rationale: matches the backend's flat-folder model (`FolderService.cs:20-33`); adding a tree would need a schema change (Phase 8+).

### 3.3 Dashboard note list: Pinned + Recent sections

- The middle pane (`<NoteList>`) renders two `<section>` blocks, in this order:
  1. **Pinned** — `useNotes({ isPinned: true })`, ordered by `updatedAt desc` (the backend's secondary sort), capped at **6** items. Each item is a `<NoteCard>`.
  2. **Recent** — `useNotes({})` (no filter), ordered by `updatedAt desc` (the backend's default), capped at **20** items. Each item is a `<NoteCard>`.
  3. If a folder or tag filter is active, the two sections collapse into a single **"Notes"** section that shows the filtered list (no cap on the section header, but the backend caps the result at 200 per `NoteService.cs:52`).
- The Pinned section header shows a pin icon + "Pinned (n)"; the Recent section header shows "Recent". When the filter is active, the section header is just "Notes (n)".
- The middle pane has a header bar with the "New note" button (right-aligned, primary), a "Pinned" toggle (left of "New note"; when ON, the list shows only pinned items regardless of section; off by default), and the active filter chip ("In: <folder> ×" or "Tagged: <tag> ×") that clears the filter.
- Empty state: a centered card "No notes yet. Create your first note." with a primary "New note" button.
- Loading state: 4 `<Skeleton>` cards per section.
- Error state: a centered `<Alert variant="destructive">` with the error message and a "Retry" button.

### 3.4 Note detail page: `/notes/:id`

- A read-only render of the note. Layout:
  - **Title row** (sticky top of the pane): title (h1, large) + a row of icon buttons on the right: **pin toggle** (star outline ↔ filled), **delete** (trash), **edit** (pencil). On mobile the row collapses to a single overflow menu.
  - **Metadata strip**: folder name (text + folder icon, or "Unfiled"), tag chips (each clickable → filter by that tag), updated-at relative time ("3 hours ago" via `Intl.RelativeTimeFormat`, refresh every 60 s).
  - **Content**: a `<ReadOnlyEditor>` that mounts the same TipTap extensions with `editable: false`. Renders the saved `contentJson`. **XSS contract (carried from Phase 4 R2 / `PHASE5_PLAN.md` §1 OOS)**: TipTap renders the JSON via its own DOM, which is **safe by construction** (every node is a typed component, no `dangerouslySetInnerHTML`). Never use `innerHTML` or `dangerouslySetInnerHTML` on `contentJson`.
- A "← Back to notes" link at the top.
- A "Delete" button opens `<DeleteNoteDialog>` (shadcn `AlertDialog`); on confirm, calls `useDeleteNote(id)`, optimistic removes from cache, navigates to `/`.
- A "Pin" button calls `useTogglePin(id)`, optimistic updates the cache.
- If the note is not found (`useNote(id)` returns 404), navigate to `/` with a toast: "Note not found." (handles the case where the user navigates to a deleted note via URL).

### 3.5 Note editor: `/notes/new` and `/notes/:id/edit`

- The two routes share the `<NoteEditorPage>` component. The difference is the `id` URL param:
  - `/notes/new` → `useCreateNote` mutation; submit creates a note, then `navigate('/notes/' + newId, { replace: true })`.
  - `/notes/:id/edit` → `useNote(id)` to load; `useUpdateNote(id)` mutation; submit updates, then `navigate('/notes/' + id)`.
- Layout (single column, max-width 768 px, centered):
  - **Top bar**: a "← Cancel" link (navigates back, no save), the title input (large, borderless — shadcn `input` with a `variant="ghost"` style), a "Save" button (primary) on the right. `Cmd/Ctrl+S` and `Cmd/Ctrl+Enter` shortcuts also save.
  - **Toolbar** (sticky, below the title): Bold, Italic, Strike, H1, H2, H3, BulletList, OrderedList, CodeBlock, Link (prompt for URL). Each button is a toggle (pressed state when active in the selection). Use lucide icons (`Bold`, `Italic`, `Strikethrough`, `Heading1`, `Heading2`, `Heading3`, `List`, `ListOrdered`, `Code`, `Link`).
  - **Content area**: a `<RichTextEditor>` (TipTap) with `StarterKit` + `Link` extensions. `editable: true`. Min height `60vh`. Focus ring on the outer div (TipTap's contenteditable has no built-in focus ring).
  - **Metadata footer**: a 2-column grid (left: folder picker; right: tag input) + a pin toggle. The footer is a `<Card>` with `bg-muted/40`.
- Client-side validation:
  - `title` is required (zod `min(1).max(200)`); show inline error under the input on blur.
  - `contentJson` is always a valid JSON object (TipTap guarantees this); backend validation is the second line of defence (`ContentJson` is `[Required, StringLength(262144)]` per `CreateNoteRequest.cs:13`).
  - `folderId` is optional; `tagIds` is optional.
  - `isPinned` is a boolean, default `false`.
- On submit, the client computes `contentText` by stripping HTML (a small `stripHtml(html)` helper — regex-based, server is the source of truth and will overwrite if the client value is empty).
- The `Save` button shows a `<Loader2 className="animate-spin" />` when `isSubmitting`. Inputs are disabled while submitting.
- "Cancel" with unsaved changes opens a confirmation: "Discard changes? Your edits will be lost." (shadcn `AlertDialog`).

### 3.6 TipTap configuration

- Extensions: `@tiptap/starter-kit` (Paragraph, Bold, Italic, Strike, Code, CodeBlock, Heading, BulletList, OrderedList, ListItem, Blockquote, HardBreak, HorizontalRule, History, Document, Text) + `@tiptap/extension-link` (URL prompts via `window.prompt` — shadcn-friendly, no extra modal).
- No `@tiptap/extension-image` (out of scope per `docs/PLAN.md` §12).
- No `@tiptap/extension-placeholder` (we use a visible `<textarea>`-like feel by leaving the content area empty; the user can type immediately).
- The editor's output JSON shape is the TipTap default: `{ type: "doc", content: [{ type: "paragraph", content: [{ type: "text", text: "…" }] }, …] }`. This is what `NoteService.CollectText` (line 294) walks for `contentText` derivation.
- The `editable` flag is the only difference between the editor and the read-only view; we use a `<RichTextEditor editable={true|false} initialContent={…} onUpdate={…} />` component.
- **SSR safety**: TipTap's `useEditor` returns `null` on the first render (the editor mounts in a `useEffect`). The component handles the `editor === null` case with a `<Skeleton className="h-60" />` placeholder. This avoids the React 18 strict-mode double-mount warning.
- **No `dangerouslySetInnerHTML` anywhere**. TipTap's `editor.getHTML()` is **only** used inside a TipTap-managed DOM (the editor renders it into a `<div ref={editorRef} />`); we never pass the HTML to `dangerouslySetInnerHTML`.

### 3.7 Server error mapping (notes / tags / folders)

- File: `client/src/lib/noteErrors.ts` (new), modelled on `lib/authErrors.ts:59-115`.
- API:
  ```ts
  export type NoteServerErrorKind =
    | 'VALIDATION_ERROR'      // field-level (set via form.setError for title)
    | 'NOTE_NOT_FOUND'
    | 'FOLDER_NOT_FOUND'
    | 'TAG_NOT_FOUND'
    | 'INVALID_CONTENT_JSON'
    | 'TAG_EXISTS'
    | 'FOLDER_EXISTS'
    | 'NETWORK_ERROR'
    | 'UNKNOWN_ERROR';

  export interface NoteServerError {
    kind: NoteServerErrorKind;
    message: string;
  }

  /**
   * Map an `ApiError` to either a top-level `NoteServerError` (returned to
   * the caller for toast rendering) or, for `VALIDATION_ERROR` on a form
   * with a `title` field, a field-level error via `form.setError`.
   *
   * `logEvent` is the logger tag (`note_create_failed`, etc.).
   */
  export function mapNoteError(
    err: unknown,
    form: ErrorMappingForm<{ title: string }> | null,
    logEvent: string,
  ): NoteServerError;
  ```
- The mapper is the single source of truth for error → user-message translation; the mutation hooks and the detail page call it and surface the result via `toast.error(message)` (or via `form.setError` for `VALIDATION_ERROR` on `title`).
- Validation field mapping: PascalCase `Title` → camelCase `title` (same `toCamelCase` helper as `authErrors.ts:42-45`); PascalCase `ContentJson` is **not** mapped to a form field (we trust TipTap's output, and the only place `ContentJson` could fail validation is the 262 144-char length, which is a server-side error toast).

### 3.8 Query / mutation hook architecture

- File: `client/src/hooks/notes.ts` (new) — all note-related hooks in one file (small enough to fit; ~200 lines).
- File: `client/src/hooks/tags.ts` (new) — `useTags`, `useCreateTag`, `useDeleteTag`.
- File: `client/src/hooks/folders.ts` (new) — `useFolders`, `useCreateFolder`, `useDeleteFolder`.
- Hook signatures:
  ```ts
  // notes
  export function useNotes(filter: NotesFilter = {}): UseQueryResult<NoteResponse[]>;
  export function useNote(id: string | undefined): UseQueryResult<NoteResponse>;
  export function useCreateNote(): UseMutationResult<NoteResponse, ApiError, CreateNoteRequest>;
  export function useUpdateNote(id: string): UseMutationResult<NoteResponse, ApiError, UpdateNoteRequest>;
  export function useDeleteNote(): UseMutationResult<void, ApiError, string>;
  export function useTogglePin(): UseMutationResult<NoteResponse, ApiError, string>;
  // tags
  export function useTags(): UseQueryResult<TagResponse[]>;
  export function useCreateTag(): UseMutationResult<TagResponse, ApiError, CreateTagRequest>;
  export function useDeleteTag(): UseMutationResult<void, ApiError, string>;
  // folders
  export function useFolders(): UseQueryResult<FolderResponse[]>;
  export function useCreateFolder(): UseMutationResult<FolderResponse, ApiError, CreateFolderRequest>;
  export function useDeleteFolder(): UseMutationResult<void, ApiError, string>;

  export interface NotesFilter {
    folderId?: string | null;   // null = unfiled; undefined = any
    tagId?: string;
    isPinned?: boolean;
    limit?: number;
  }
  ```
- Query keys: `['notes', filter]` (object so cache is per-filter); `['notes', id]`; `['tags']`; `['folders']`. The mutation hooks `invalidateQueries(['notes'])` and (when relevant) `['tags']` / `['folders']` on `onSettled` (so a stale list refreshes after a mutation).

### 3.9 Optimistic update strategy

- **Pin toggle** (`useTogglePin`): `onMutate` cancels the `['notes']` queries, snapshots the previous cache, patches the matching note's `isPinned` via `setQueryData`, `onError` rolls back from the snapshot, `onSettled` invalidates `['notes']`. The detail page and the list both reflect the change instantly; the server round-trip is the source of truth (the `onSettled` invalidate re-fetches the canonical state).
- **Delete** (`useDeleteNote`): `onMutate` cancels + snapshots, removes the note from every `['notes', filter]` cache entry, `onError` rolls back, `onSettled` invalidates `['notes']` and `['folders']` (because `noteCount` changes).
- **Update** (`useUpdateNote`): `onMutate` cancels + snapshots, replaces the note in the `['notes', id]` cache and in every list cache that contains it, `onError` rolls back, `onSettled` invalidates `['notes']`.
- **Create** (`useCreateNote`): **not optimistic** — the create form is modal-ish, the user expects the "Save" button to mean "actually saved", and a rollback UI for a note that doesn't yet exist on the server would be confusing. We show a loading spinner and `onSuccess` navigate.
- **Tag/folder create** (inline from the tag input / sidebar): **not optimistic** either, because the create flow has its own UI (a popover for the new tag name, a dialog for the new folder name) and the user is expecting a new option to appear in the picker.
- All mutations log `event_attempt` / `event_succeeded` / `event_failed` via `logger.info` / `logger.warn` (only the `code` is logged on failure, never the body — same precedent as `lib/authErrors.ts:86`).

**Conventions to follow (match Phase 4 + 5):**

- TypeScript strict + `noUncheckedIndexedAccess`.
- File-scoped imports, `import type` for types.
- No `default` exports except where shadcn requires them (the new shadcn components use default exports; that's shadcn's convention).
- PascalCase components; camelCase hooks; `useX` for hooks, `XDialog` for dialog wrappers, `XButton` for button components.
- TanStack Query: `queryKey` is a tuple (e.g. `['notes', filter]`); the filter object is created with `useMemo` so it's referentially stable across renders.
- Optimistic updates: `onMutate` → `cancelQueries` → `setQueryData` (with the previous data snapshot) → return the snapshot. `onError` → `queryClient.setQueryData(key, snapshot)`. `onSettled` → `invalidateQueries(key)`.
- No `dangerouslySetInnerHTML` anywhere (the Phase 3 R2 / Phase 4 R2 XSS contract). TipTap renders its own DOM.
- All API calls go through `api/client.ts`. No raw axios.
- Logger fires `note_list_requested`, `note_list_failed`, `note_get_failed`, `note_create_attempt`, `note_create_succeeded`, `note_create_failed`, `note_update_attempt`, `note_update_succeeded`, `note_update_failed`, `note_delete_attempt`, `note_delete_succeeded`, `note_delete_failed`, `note_pin_toggle_attempt`, `note_pin_toggle_succeeded`, `note_pin_toggle_failed`, `tag_create_attempt`, `tag_create_succeeded`, `tag_create_failed`, `folder_create_attempt`, `folder_create_succeeded`, `folder_create_failed`, `folder_delete_attempt`, `folder_delete_succeeded`, `folder_delete_failed`. Names mirror the backend's Serilog events (`note_created`, `note_updated`, `note_deleted`, `note_pin_toggled`, `tag_created`, `tag_deleted`, `folder_created`, `folder_deleted` — per `NoteService.cs:115-117,186-188,217-219,239-241`, `TagService.cs:54-57,72-75`, `FolderService.cs:57-60,75-78`).
- Every mutation's `onError` calls `mapNoteError(err, form, logEvent)` and the returned `message` is rendered via `toast.error(message)`. The mapper is the single source of truth for the user-facing message.
