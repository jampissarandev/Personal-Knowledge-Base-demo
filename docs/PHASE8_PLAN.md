### 3.4 Backend log audit: 7-day window, threshold = 1 occurrence

- The audit covers `Logs/app-20260619.log` through the most recent `app-*.log`.
- For each `[ERR]` / `[FTL]` line, the auditor (sub-agent) reads the line + the 5 preceding context lines + the URL/method/exception, then classifies:
  - **Repeated 5xx** (3+ occurrences of the same `Exception` type or the same `URL`) → **bug**, file a Critical in `PHASE8_PLAN.md` §9.
  - **Repeated 4xx** (5+ occurrences of the same `URL` returning 401) → **expected** (token expiry), file a Nit.
  - **Single 5xx** → **paper cut**, file a Nit.
  - **Uncaught exception** (anything not wrapped in the `ExceptionHandlingMiddleware` envelope) → **bug**, file a Critical.
- The audit produces a markdown table: `Time | Level | URL | Exception | Classification | Notes`.
- The output lives in `docs/PHASE8_PLAN.md` §5.5 (the Verification checklist).
- Rationale: production-grade log auditing is a 1-day exercise; the 7-day window is the entire project history so far, so the audit is a one-shot pass, not an ongoing concern.

### 3.5 Final security audit: the 7-point checklist from `PLAN.md` §7

- The 5-axis review from `code-review-and-quality` is run by 3 parallel sub-agents per the orchestration pattern (`code-reviewer` + `security-auditor` + `test-engineer`).
- **Security-specific checklist** (cross-referenced with `PLAN.md` §7):
  1. **Password hashing** — `Program.cs` configures `IPasswordHasher<ApplicationUser>` (ASP.NET Identity default, PBKDF2 with SHA-256, 100k iterations, 16-byte salt). Verify by reading `Program.cs:86-96` + `appsettings.json`.
  2. **JWT signing key** — `appsettings.json` has the placeholder `__production_warning__` comment; `appsettings.Development.json` has a 64-char dev key. Verify the prod key handling (see §3.7 — secrets management).
  3. **HTTPS redirect** — `Program.cs` line ~140 has `app.UseHttpsRedirection()` (or equivalent). Verify it's enabled in `Production` env, NOT in `Development` (so the dev proxy works over HTTP).
