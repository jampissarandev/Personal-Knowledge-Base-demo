### Deliverable

A scaffolded `client/` application that:

- Boots in dev mode (`npm run dev`) and prod build (`npm run build`) with **0 warnings, 0 errors**.
- Mounts a router with at least four routes: `/login`, `/register`, `/`, and `*` (NotFound). Only `/login` and `/register` are public; everything else is gated.

- Handles 401 globally by clearing the token and redirecting to `/login`.
- Renders a `<Layout>` (header + outlet) with a working dark-mode toggle (next-themes).
- Logs every API call (`method url → status`) and every error to `console` via `lib/logger.ts`.
- Exposes a typed `api` module that Phase 5–7 will extend (`auth.ts`, `notes.ts`, `tags.ts`, `folders.ts`, `search.ts` — folders in this phase, others deferred).

### 3.2 Tailwind v3, not v4

- shadcn v3 templates are written for v3 (`tailwind.config.js` + `postcss.config.js` + `@tailwind base; @tailwind components; @tailwind utilities;`).
- Tailwind v4 uses `@import "tailwindcss";` and CSS-side theme tokens. Mixing shadcn v3 with v4 requires shadcn upgrade + a migration pass; out of scope for foundation.
- Re-evaluate at end of Phase 7 if shadcn v4 stabilises.

### 3.3 Vite dev-server proxy for `/api`

`vite.config.ts` will proxy `/api` → `http://localhost:5000`:

```ts
server: {
  port: 5173,
  proxy: {
    '/api': {
      target: 'http://localhost:5000',
      changeOrigin: true,
    },
  },
},
```

- Avoids CORS preflight round-trips in dev.
- Same-origin in production once both are behind a single host.
- Note: CORS policy in `Program.cs` is still required for cases where the dev port differs (e.g. curl from Postman).

### 3.4 Token storage — `localStorage`

- PLAN.md §7 does not specify. The backend returns the JWT in the response body; HttpOnly cookies would require a CSRF strategy and a separate auth endpoint. Phase 4 picks `localStorage` and accepts the XSS trade-off for a personal, single-user app.
- The `SearchHit` snippet XSS mitigation (Phase 3 R2) is unaffected: snippets are rendered as text, never `innerHTML`.
- Documented in `lib/tokenStorage.ts` JSDoc.

### 3.9 Auth context shape

```ts
type AuthContextValue = {
  user: UserResponse | null;
  token: string | null;
  isAuthenticated: boolean;
  isLoading: boolean;          // true while /api/auth/me is in-flight on mount
  login: (token: string, user: UserResponse) => void;
  logout: () => void;
};
```

`<AuthProvider>` initialisation:
1. Read token from localStorage.
2. If present, call `GET /api/auth/me` in the background; on success, set `user`. On 401, clear the token.
3. `isLoading` is `true` until step 2 resolves. `<ProtectedRoute>` shows a spinner while `isLoading` is true (avoids the "flash of redirect" race).

### 3.10 Logger surface

```ts
export const logger = {
  info:  (msg: string, meta?: Record<string, unknown>) => console.info(`[INFO] ${msg}`, meta),
  warn:  (msg: string, meta?: Record<string, unknown>) => console.warn(`[WARN] ${msg}`, meta),
  error: (msg: string, err?: unknown, meta?: Record<string, unknown>) => console.error(`[ERROR] ${msg}`, err, meta),
  api:   (method: string, url: string, status?: number, elapsedMs?: number) =>
    console.log(`[API] ${method} ${url} → ${status ?? "pending"} (${elapsedMs ?? "?"}ms)`),
};
```

Matches the spec in `docs/PLAN.md` §5 with the addition of `elapsedMs` (timing is the only meaningful metadata an axios interceptor can capture uniformly). Wire `api` into the response interceptor.

**Conventions to follow:**

- **TypeScript strict**, `noUncheckedIndexedAccess: true`.
- File-scoped **no semicolon** is fine (Vite default for `src/`).
- Components: arrow-function components for small ones, function declarations for pages and providers.
- Hooks: `useAuth`, `useLogin`, `useTheme` are the only Phase 4 hooks.
- All API calls go through `api/client.ts`; never instantiate a raw axios elsewhere.
- All errors from axios are caught and re-thrown as `ApiError(code, message, status)` so components can `instanceof ApiError` to discriminate.
- Use TanStack Query only for `/api/auth/me` in this phase. Other endpoints are stub functions returning empty arrays; the stubs are wired in later phases.
- No `any`. Prefer `unknown` + narrowing. (Phase 4 has very few unknowns because the API surface is small.)
- `<Outlet />` from `react-router-dom`, not the `children` prop on `Layout`.
