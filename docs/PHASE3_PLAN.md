**FTS5 design (binding for the implementer):**

Use a **contentless FTS5 table** to avoid double-storing text and to keep the FTS index
in sync via triggers on the `Notes` table:

```sql
-- Virtual table (contentless, external content = Notes)
CREATE VIRTUAL TABLE notes_fts USING fts5(
    title,
    content_text,
    content='Notes',
    content_rowid='rowid',
    tokenize='porter unicode61'
);

-- Initial backfill (idempotent)
INSERT INTO notes_fts(rowid, title, content_text)
    SELECT rowid, Title, ContentText FROM Notes;

-- Sync triggers
CREATE TRIGGER notes_fts_ai AFTER INSERT ON Notes BEGIN
    INSERT INTO notes_fts(rowid, title, content_text)
    VALUES (new.rowid, new.Title, new.ContentText);
END;

CREATE TRIGGER notes_fts_ad AFTER DELETE ON Notes BEGIN
    INSERT INTO notes_fts(notes_fts, rowid, title, content_text)
    VALUES ('delete', old.rowid, old.Title, old.ContentText);
END;

CREATE TRIGGER notes_fts_au AFTER UPDATE ON Notes BEGIN
    INSERT INTO notes_fts(notes_fts, rowid, title, content_text)
    VALUES ('delete', old.rowid, old.Title, old.ContentText);
    INSERT INTO notes_fts(rowid, title, content_text)
    VALUES (new.rowid, new.Title, new.ContentText);
END;
```

Notes:
- `content='Notes'` + `content_rowid='rowid'` makes FTS5 read the indexed columns from
  `Notes` at query time. This avoids duplicating storage.
- `tokenize='porter unicode61'` is the standard analyzer (English stemming, Unicode).
  No language-specific config needed.
- The `'delete'` command in the AD/AU triggers is the FTS5 "external content" delete
  syntax — without it, deletes silently leave stale rows.
- `rowid` is the hidden SQLite rowid; `Notes.Id` is a separate GUID. The FTS5 rowid
  is mapped to `Notes.rowid` by joining `Notes.rowid = notes_fts.rowid`. This means
  the FTS5 rowid is **not** stable across migrations that rewrite the table — but
  since we never expose rowid externally, this is fine.
- Trigger `notes_fts_ad` uses `'delete'` so FTS5's external-content mirror is purged.

**Migration: the FTS5 schema must be added in raw SQL via `migrationBuilder.Sql(...)`**
because EF Core cannot model FTS5 virtual tables. The migration must be applied with
`dotnet ef database update` (no model-snapshot changes — FTS5 is not in the EF model).

**Search query strategy (binding):**

The FTS5 query must be **owner-scoped** because FTS5 itself does not enforce `UserId`.
The cleanest pattern is a `WHERE MATCH(?)` on `notes_fts` joined to `Notes` with a
`UserId` filter:

```sql
SELECT n.Id, n.Title,
       snippet(notes_fts, 1, '<mark>', '</mark>', '…', 32) AS Snippet,
       bm25(notes_fts) AS Rank
FROM   notes_fts
JOIN   Notes n ON n.rowid = notes_fts.rowid
WHERE  notes_fts MATCH @q
       AND n.UserId = @userId
ORDER  BY Rank
LIMIT  @limit OFFSET @offset;
```

This query must run via `_context.Database.SqlQueryRaw<...>(...)` or via
`FromSqlRaw` on a keyless entity. **Do NOT use `LIKE`** — the whole point of this
phase is FTS5.

**Query syntax handling (FTS5 user input):**

FTS5 has its own query syntax (`"phrase"`, `term*`, `NEAR(a b)`, `AND/OR/NOT`, column
filters `title:foo`). Two acceptable strategies, listed in preference order:

1. **Quote the user input verbatim.** Wrap the entire `q` in double quotes and escape
   any inner double quotes by doubling them: `q.Replace("\"", "\"\"")` then
   `"\"" + escaped + "\""`. This treats the input as a phrase search — safe and
   predictable.
2. Pass raw. Risky (users can inject `AND/OR/NOT` to manipulate ranking) — not recommended.

This plan mandates **strategy 1** (phrase search by default). If the future wants
operator support, add an explicit `mode=or|and|phrase` query parameter in a later phase.

**Endpoint contracts (binding):**

- `GET /api/search?q=...&limit=20&offset=0`
- `q` required, `[Required, MinLength(1), StringLength(200)]` (DataAnnotations).
  Empty `q` → `400 VALIDATION_ERROR`.
- `limit` optional, default 20, clamped 1..100.
- `offset` optional, default 0, clamped 0..1000.
- `200 OK` with `SearchResponse { Q, Total, Hits[] }` where each hit is
  `SearchHit { NoteId, Title, Snippet }` (snippet is HTML-safe with `<mark>` tags;
  caller is responsible for sanitizing on render).
- Empty result set → `200` with `Total=0, Hits=[]` (not `404`).
- `401 UNAUTHORIZED` if no token.
- `500 SEARCH_ERROR` on FTS5 query failure (e.g., syntax error in the quoted phrase
  should be impossible because we wrap, but guard anyway).

**SearchResponse shape (binding):**

```csharp
public record SearchHit(Guid NoteId, string Title, string Snippet);

public record SearchResponse(
    string Q,
    int Total,
    IReadOnlyList<SearchHit> Hits);
```

## 6. Open Questions for Backend Engineer

1. **Migration approach:** Should the FTS5 migration be applied via
   `dotnet ef database update` (manual, before `dotnet run`) or auto-apply on
   startup via `app.Services.GetRequiredService<AppDbContext>().Database.Migrate()`
   in `Program.cs`? The codebase does not currently auto-apply; this plan keeps
   it manual. **Decision: manual via `dotnet ef database update`**, then
   document in README. If the team wants auto-apply, that's a small follow-up.

2. **Mapster for SearchResponse:** Phase 2 uses Mapster for `Note → NoteResponse`.
   Should `SearchHit` use Mapster? **Decision: no** — `SearchHit` is built directly
   from the FTS5 query result. This avoids an extra `.Adapt<>()` call and matches
   the pattern of `FolderService.CreateAsync` (which builds `FolderResponse`
   manually, see Phase 2 §6 N3).

3. **Logging `q` content:** The `search_query` log event will record
   `qLength=q.Length, qHash=SHA256(q)` (not the raw `q`). This avoids PII
   leakage while still letting ops debug "is the user searching for empty
   strings?" and "are queries always < 200 chars?".

4. **Connection string `Cache=Shared`:** Some FTS5 guides recommend
   `Data Source=app.db;Cache=Shared` to allow the FTS5 module to coexist with
   EF Core's pooled connection. **Decision: leave as `Data Source=app.db`**.
   FTS5 works with the default pooled connection in `Microsoft.Data.Sqlite 8+`.

### 8.3 Architectural Decisions (binding deviations from §3 step A)

1. **FTS5 table variant = `regular` (no `content` option).** The plan's §3 step A
   specified `content='Notes'` with a comment that called it "contentless". Those
   two are mutually exclusive FTS5 modes. After experimentation:
   - `content='Notes'` (external-content): rejects direct INSERT (the AI trigger
     fails with "SQL logic error"), the plan's backfill SQL is invalid for this
     variant, and the `'delete'` command in AD/AU triggers does not work because
     there is no external content to invalidate.
   - `content=''` (contentless): snippet()/bm25() are unavailable, so the
     response contract cannot be honoured.
   - **Regular FTS5** (no `content` clause, content stored inside the FTS5
     shadow tables): direct INSERT/DELETE/REPLACE work, snippet() works,
     bm25() works, the plan's trigger shape is valid. Storage cost is ~2× the
     indexed columns — acceptable for personal note sizes.
   - **Decision:** regular FTS5. Documented in the migration file's comments.

2. **Trigger syntax = regular `DELETE` + `INSERT`, not the FTS5 `'delete'`
   command.** The plan's AD/AU triggers used the FTS5 `'delete'` command
   (`INSERT INTO notes_fts(notes_fts, rowid, …) VALUES ('delete', …)`), which
   is valid only for contentless/external-content tables. With a regular
   FTS5 table this raises "SQL logic error" on every UPDATE/DELETE, which
   surfaced as 500s on the existing Notes endpoints. Replaced with:
   ```sql
   -- AD
   DELETE FROM notes_fts WHERE rowid = old.rowid;
   -- AU
   DELETE FROM notes_fts WHERE rowid = old.rowid;
   INSERT INTO notes_fts(rowid, title, content_text) VALUES (new.rowid, …);
   ```

3. **Implementation path = raw `SqliteConnection` via `GetDbConnection()`,
   not EF `SqlQueryRaw<T>` / `FromSqlRaw`.** The plan allowed both. We tried
   `Database.SqlQuery<FtsSearchRow>(FormattableString)` first; it produced
   `SQLite Error 1: 'no such column: T.content_text'` because EF's
   `SqlQuery` rewriting aliases the projection in a way the FTS5 `snippet()`
   function does not survive. Raw ADO.NET bypasses all EF query rewriting
   and executes the exact SQL we wrote. Reference: `SearchService.cs:33-83`.

4. **Tokenization unchanged** from the plan: `tokenize='porter unicode61'`
   (English stemming + Unicode normalization).

5. **Phrase-search quoting** unchanged from the plan:
   `q.Replace("\"", "\"\"")` then wrap in `"…"`. Inner double-quotes are
   doubled; FTS5 cannot interpret operator tokens because everything is
   inside the phrase string.

6. **Logging fields added per the plan's §6 Q3:** `qLength`, `qHash` (first
   4 bytes of SHA-256 in hex), `resultCount`, `total`, `elapsedMs`.
   Raw `q` is **never** written to the log — only its length and a short
   hash for debugging without PII leakage.
