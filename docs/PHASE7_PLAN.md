### Backend search contract (binding for §3.4)

| Parameter | Source | Constraint |
|---|---|---|
| `q` (required) | `SearchController.cs:30` | 1..200 chars; rejected by DataAnnotations if empty or too long |
| `limit` (optional) | `SearchController.cs:31`, clamp `1..100`, default `20` | per-page cap for `/search` is 20 (matches Phase 7 §3.6 pagination) |

Response shape (`SearchResponse`):
```ts
{
  query: string;        // echo of the q parameter (trimmed)
  total: number;        // total matching rows, across all pages
  hits: SearchHit[];    // ordered by FTS5 rank desc, then updatedAt desc
}

interface SearchHit {
  noteId: string;       // Guid
  title: string;        // note title (no HTML, plain text)
  snippet: string;      // server-derived excerpt; may contain literal <mark>…</mark> tags wrapping the match
  score: number;        // FTS5 rank (lower = better match in FTS5, but server returns the negation if needed)
}
```

The `snippet` format is backend-defined (`SearchService.cs` line ~140 emits `<mark>match</mark>` around the matched term in the surrounding context). The frontend MUST treat the snippet as text — see §3.5 for the render contract.

## 3. Binding Decisions

These pin the spec so Step A has no ambiguity. Each deviation from `docs/PLAN.md` §8 or §10 is called out. **If you disagree with any of these, say so before Step A starts.**

### 3.1 Search bar: header-mounted, debounced 250 ms, dropdown of 5 results

- The search bar lives in the **header** (between the PKB logo and the theme toggle), as a single `<Input>` with a `Search` icon (lucide) on the left and an `X` clear button on the right (only visible when the input has a value).
- Width: `flex-1 max-w-md` (the header's `container` is the same as Phase 4's; the search bar fills the available space up to 28 rem).
- Debounce: **250 ms** after the last keystroke. Below 250 ms the network feels chatty; above 400 ms it feels laggy. 250 ms is the GitHub / Linear / Notion sweet spot.
- **In-flight cancellation**: every new keystroke calls `controller.abort()` on the previous `AbortController` (the `search` function takes an optional `signal: AbortSignal` argument; `useSearch` creates a new `AbortController` per query and aborts the previous one on each new debounce fire). The backend treats an aborted request as a no-op (its `CancellationToken` is plumbed through to `_searchService.SearchAsync`).
- **Dropdown** (when the input is focused AND has at least 1 character AND at least 1 result):
  - Shows the **top 5** hits, each row = `title` (truncated to 60 chars + ellipsis) + `snippet` (text, see §3.5) + small `folderName` text (or "Unfiled") on the right.
  - Below the 5 rows, a separator and a "Search all results for `<q>`" button (submits to `/search?q=…`).
  - Below that, if `total === 0`, an empty-state row: "No matches for `<q>`" + a "Create note titled `<q>`" link (navigates to `/notes/new?title=<encoded-q>`).
  - Closes on `Escape`, on click outside, on route change.
  - Keyboard: `ArrowDown` / `ArrowUp` move the highlight, `Enter` on a highlighted row navigates to `/notes/<id>`, `Enter` on the "Search all" button navigates to `/search?q=<q>`.
- The dropdown is a shadcn `<Popover>` anchored to the input. It uses the existing `popover.tsx` primitive (no new shadcn component needed).
- **Mobile** (< 768 px): the search bar collapses to a `Search` icon button in the header (between the PKB logo and the theme toggle). Clicking it opens a shadcn `<Sheet>` (already installed) that contains the full-width input + the dropdown rendered inside the sheet. Closing the sheet clears the input. This is the same pattern as the `<MobileFolderSheet>` (Phase 6).
- **`/` keyboard shortcut**: when the user is **not** typing in another input (`<input>`, `<textarea>`, or TipTap's contenteditable), pressing `/` focuses the search bar. Implemented with a global `keydown` listener on `document`, registered in `SearchBar.tsx` and torn down on unmount. Documented in the component JSDoc because it's a non-obvious behaviour.
- Rationale: matches Notion / GitHub / Linear; debounce + abort is the standard pattern; 5-result dropdown gives instant feedback without clutter; "Search all" is the escape hatch.

### 3.2 `/search` route: paginated results, same `<NoteCard>` shape

- Route: `/search?q=<encoded>&offset=<n>`, protected by `<ProtectedRoute>` (the existing one is fine; `?q=` is not a `next=` redirect).
- Layout (single column, max-width 768 px, centered — matches the editor's max-width):
  - **Header strip**: the search input (pre-filled with `q`, with the same debounce + dropdown as the header bar; typing in it navigates to the new `?q=`). Below it, a count line: "**N** results for **<q>**" (or "1 result" / "0 results" — proper pluralisation).
  - **Result list**: each row is a `<SearchResultRow>` (new) — clickable, navigates to `/notes/<id>`. Renders `title` (h3), `snippet` (text, see §3.5), and a metadata strip (`folderName` + tag chips + relative updated-at). Hover state: subtle `bg-muted/40`. Focus ring on `focus-visible`.
  - **Pagination**: a "Load more" button at the bottom. On click, calls `useSearch({ q, offset: hits.length })` and **appends** the new hits to the existing list. Button is disabled while loading; hidden when `hits.length >= total`.
  - **Empty state**: when `total === 0` and no error, render a centered card: "No matches for **<q>**" + a primary "Create note titled **<q>**" button (navigates to `/notes/new?title=<encoded-q>`).
  - **Loading state**: 5 `<Skeleton>` rows on the initial load (before the first response); a small inline `<Loader2 className="animate-spin" />` next to the "Load more" button on subsequent loads.
  - **Error state**: a centered `<Alert variant="destructive">` with the error message and a "Retry" button.
- The query is in the URL, not in component state — the back button works, the result page is shareable.
- Rationale: the `?q=` URL is the source of truth; the `<NoteCard>` reuse means the result → detail navigation is identical to the dashboard; pagination is server-driven (no client-side slicing of a giant list).

### 3.3 Search hook: `useSearch` with abort + debounce

- File: `client/src/hooks/search.ts` (new).
- Signatures:
  ```ts
  export interface UseSearchParams {
    q: string;                       // already debounced + trimmed; empty string disables the query
    limit?: number;                  // default 20
    offset?: number;                 // default 0; used for "Load more"
    enabled?: boolean;               // default true
  }

  /**
   * TanStack Query wrapper around `searchApi.search`. When `q` is empty,
   * the query is disabled (no request fires). The `signal` argument
   * propagates TanStack's internal AbortSignal, which is the source of
   * truth for in-flight cancellation.
   */
  export function useSearch(
    params: UseSearchParams,
  ): UseQueryResult<SearchResponse, ApiError>;
  ```
- Debounce is **not** inside `useSearch`; it lives in `<SearchBar />` (a `useEffect` with `setTimeout(..., 250)` and a cleanup that clears the timeout). The hook receives the **already-debounced** `q` from the search bar; the `/search` page reads `q` from `useSearchParams` (no debounce — the URL is the source of truth, navigating to `?q=foo` is an explicit user action).
- Query key: `['search', q, limit, offset]`. Two different offsets for the same `q` are **separate** cache entries (TanStack handles the parallel fetch); on "Load more" the hook is called with a new `offset` and the new result is **appended** in the consumer (not in the hook).
- `staleTime`: 30 s for search results (search results are not time-critical; 30 s saves a round-trip on the back button).

    offset = 0,

### 3.5 Snippet render contract: text-only, no `dangerouslySetInnerHTML`

- The `snippet` from the backend contains literal `<mark>match</mark>` tags. The render contract is:
  1. **Never** pass the snippet to `dangerouslySetInnerHTML` (this is the Phase 3 R2 + Phase 4 R2 + Phase 6 §3.6 carryover).
  2. The simplest correct render is: render the snippet as a plain string (React escapes `<` and `>`), and visually de-emphasise the literal `<mark>` and `</mark>` so the user sees them as text (e.g. `text-muted-foreground`). This is the "honest text" approach — the user sees the raw server output.
  3. The "pretty" approach: parse the snippet with a small `parseSnippet(snippet): Array<{ text: string; mark: boolean }>` helper that splits on the literal `<mark>` / `</mark>` tags and renders the marked runs as `<mark>` elements with a Tailwind class (`bg-yellow-200 text-foreground dark:bg-yellow-900`).
- **Phase 7 picks option 3** (the "pretty" approach) because it matches the user's mental model ("the matched text should be highlighted, not shown as `<mark>`"). The helper is:
  ```ts
  /**
   * Parse a server-issued snippet (which wraps matches in literal
   * `<mark>…</mark>` tags) into a list of renderable runs. The
   * output is intended to be rendered with React's automatic
   * escaping — never via `dangerouslySetInnerHTML`.
   *
   * The parser is intentionally simple (no HTML entity decoding
   * beyond the tags we know about). The server is the source of
   * truth; if the server starts emitting a different tag set, this
   * helper will fall back to rendering the snippet as a single
   * unmarked run.
   */
  export function parseSnippet(snippet: string): Array<{ text: string; mark: boolean }>;
  ```
- The helper is in `client/src/lib/parseSnippet.ts` (new, ~20 lines, pure function, easy to test by hand). The consumer is `<SearchResultRow>` (in the `/search` page) and `<SearchBarDropdownRow>` (in the header dropdown).
- **XSS review (binding for Step B security-reviewer):** the helper never calls `dangerouslySetInnerHTML`, never uses `innerHTML`, and never uses `eval`. The output runs are rendered with React's automatic escaping (`<span>{run.text}</span>`). The literal `<mark>` and `</mark>` strings in the snippet are matched with `String.prototype.split` (not a regex on HTML, not a parser), so a snippet containing malformed tags (e.g. `<mark>unclosed`) is gracefully handled: the unmatched `<mark>` is rendered as a literal string in the unmarked run, the matches inside are still highlighted.
- Rationale: highlights match user expectation; the helper is small and provably safe; the contract is explicit and documented.

### 3.6 `mapSearchError` + unified error surface

- File: `client/src/lib/searchErrors.ts` (new), modelled on `lib/noteErrors.ts` (Phase 6) and `lib/authErrors.ts` (Phase 5).
- API:
  ```ts
  export type SearchServerErrorKind =
    | 'VALIDATION_ERROR'              // q too long (> 200 chars) — toast, not a form field
    | 'SEARCH_QUERY_TOO_MANY_TOKENS'  // q has too many words
    | 'SEARCH_ERROR'                  // 500 from the FTS5 engine
    | 'NETWORK_ERROR'
    | 'UNKNOWN_ERROR';

  export interface SearchServerError {
    kind: SearchServerErrorKind;
    message: string;
  }

  /**
   * Map an `ApiError` (or unknown) to a user-facing message. Used by
   * `<SearchBar />` (toast) and `/search` page (inline alert).
   * The mapper logs the error code via `logger.error` so the
   * server-side event stream matches the backend Serilog event name.
   */

    err: unknown,
    logEvent: string,
  ): SearchServerError;
  ```
- The mapping table is the §2 "Backend error codes" table above. `UNAUTHORIZED` (401) is **not** mapped here — the `api/client.ts` interceptor clears the token and the `ProtectedRoute` guard handles the redirect; the `mapSearchError` function is only called for non-auth errors.
- The `logEvent` parameter is the logger tag (e.g. `search_requested`, `search_failed`, `search_load_more_failed`).

## 6. Risks & Open Questions

| # | Risk | Mitigation |
|---|---|---|
| R1 | The debounce timer + the AbortController can race: a rapid keystroke fires a new `useEffect`, which sets a new debounce timeout, but the previous request is still in flight. | The `useSearch` hook is keyed on `q`; when `q` changes, TanStack cancels the previous query (its built-in behaviour: a query with a new key invalidates the in-flight request for the old key). The 250 ms debounce ensures `q` only changes after the user stops typing, so the race window is small. |
| R2 | The snippet parser uses `String.prototype.split` on the literal `<mark>` / `</mark>` tags. If the backend starts emitting a different tag set (e.g. `<b>`), the helper will silently render the snippet as a single unmarked run (acceptable degradation). | The helper is pure and unit-testable; the JSDoc documents the contract. If the backend changes the tag set, the helper is updated in one place. |
| R3 | The `/` keyboard shortcut can fire on the auth pages (Login / Register) where the search bar is hidden. | The `<SearchBar />` component is rendered inside `<Header />` (Phase 4), which is rendered for **all** routes (auth + 404 + dashboard + search). The search bar is hidden on `/login` and `/register` (no `useAuth().user`), but the keyboard listener is still registered.
