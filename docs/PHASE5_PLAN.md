## 3. Binding Decisions

These pin the spec so Step A has no ambiguity. Each deviation from `docs/PLAN.md` §10 is called out. **If you disagree with any of these, say so before Step A starts.**

### 3.1 Form library: `react-hook-form` + `zod` (NOT in `docs/PLAN.md` §10)

`docs/PLAN.md` §10 doesn't specify a form library. Pick:

- **`react-hook-form` ^7.80** — the de-facto standard; uncontrolled inputs (no re-render on every keystroke), small bundle (~9 kB gzipped), great TypeScript inference, mature error-handling API.
- **`zod` ^3.23** (NOT 4.x — 4.x is new with breaking changes; RHF resolvers haven't fully caught up) — the de-facto schema validation library; first-class TS inference; matches shadcn's `Form` component which is built on `@hookform/resolvers/zod`.
- **`@hookform/resolvers` ^3.10** — the bridge that wires a zod schema into RHF via `zodResolver(schema)`.

Why not `formik`? `formik` is in maintenance mode; `react-hook-form` is the active standard. Why not hand-rolled? Phase 5 has 5 form fields with non-trivial validation; hand-rolled `useState` + manual validation is more code, more bugs, worse TS inference.

### 3.2 Toast library: `sonner` (already in deps, wire it in Phase 5)

`docs/PHASE4_PLAN.md` §3.1 noted sonner was deferred to "first toast consumer" — that's Phase 5. Use the shadcn `sonner` wrapper (`npx shadcn@latest add sonner` adds `components/ui/sonner.tsx` + a `<Toaster />`). Mount it in `main.tsx` next to `<RouterProvider>`. Wrap sonner in `lib/toast.ts` so we have one import path: `import { toast } from '@/lib/toast'`.

### 3.6 Client-side validation (zod schemas)

```ts
// lib/schemas/login.ts
export const loginSchema = z.object({
  email: z.string().email().max(256),
  password: z.string().min(1).max(100),
});
export type LoginFormValues = z.infer<typeof loginSchema>;

// lib/schemas/register.ts
const passwordRule = z.string()
  .min(8, 'Password must be at least 8 characters.')
  .max(100, 'Password must be at most 100 characters.')
  .regex(/[A-Z]/, 'Password must contain at least one uppercase letter.')
  .regex(/[a-z]/, 'Password must contain at least one lowercase letter.')
  .regex(/[0-9]/, 'Password must contain at least one digit.')
  .regex(/[^A-Za-z0-9]/, 'Password must contain at least one symbol.');

export const registerSchema = z.object({
  email: z.string().email().max(256),
  password: passwordRule,
  displayName: z.string().max(100).optional().or(z.literal('')),
});
export type RegisterFormValues = z.infer<typeof registerSchema>;
```

Why the regex rules? Phase 1's `AuthService.cs:46` uses ASP.NET Identity's default password policy. The regex set above is a stricter client-side preview of what Identity accepts; if a user passes our client check, the server will accept it. The reverse isn't true (a server `REGISTRATION_FAILED` can still come back if a future Identity policy is added), which is why §3.7 maps that code to a generic message.

`displayName` is `z.string().max(100).optional().or(z.literal(''))` to allow the form to submit an empty string (the backend treats empty as null).

### 3.7 Server error mapping

| Server code | RHF action | UI |
|---|---|---|
| `VALIDATION_ERROR` (400) | iterate `error.details` and call `setError(field, { type: 'server', message })` for each field key (e.g. `Email`, `Password` → `email`, `password` field name) | inline field errors; no top-level alert |
| `EMAIL_TAKEN` (409) | none | top-level `<Alert variant="destructive">` above the form: "An account with this email already exists. [Sign in instead](/login?next=…)." |
| `REGISTRATION_FAILED` (400) | none | top-level alert + `toast.error('Registration failed.')` |
| `INVALID_CREDENTIALS` (401) | none | top-level alert: "Invalid email or password." + `toast.error(...)` |
| `NETWORK_ERROR` (0) | none | `toast.error('Unable to reach the server. Check your connection.')` |
| `UNKNOWN_ERROR` (5xx) | none | `toast.error('Something went wrong.')` |

Why inline alert + toast? Inline alert persists on the page (user can re-read the error); toast is a one-shot notification (draws the eye). For auth flows the inline alert is the canonical surface; the toast is redundant but standard practice.

### 3.8 Redirect after login: `?next=` with open-redirect protection

- `AuthContext.login()` becomes `login(token, user, next?)`. When `next` is provided, `useNavigate()(next, { replace: true })` is called **after** the token is set.
- `next` validation: must start with `/` and **must not** start with `//` (which is a protocol-relative URL — open-redirect vector). If invalid, fall back to `/`.
- New hook `useRedirectIfAuthed()`: if `useAuth().isAuthenticated` is true, `<Navigate to={next ?? '/'} replace />` and return `null`. Use on `/login` and `/register`.
- Default `next`: `/`.

```ts
// lib/nextPath.ts
export function safeNextPath(raw: string | null): string {
  if (raw === null || raw === '') return '/';
  if (!raw.startsWith('/')) return '/';
  if (raw.startsWith('//')) return '/';
  return raw;
}
```
